## 1. Project Overview（项目基础信息）

- **项目类型**：本项目是基于 Go 1.21+ 的高性能多模块 API 服务，采用 go.work 工作空间管理多模块协同开发，主要用于归纳常用的 Golang 工具方法，方便开发人员快速使用。
- **核心约束**：本地多模块协同开发，无需修改各模块 go.mod 的 replace 指令，即可直接调用不同模块内的函数与方法。
- **禁止事项**：子模块命名需语义化、功能单一，避免"万能模块"（如禁止在 utils 模块中添加业务逻辑）。

## 2. Code Style（代码规范）

- **代码缩进**：4 个空格。
- **禁止使用**：any 类型、硬编码魔法值（如直接写 `100` 代替 `MAX_PAGE_SIZE`）。

### 2.1 命名规范

- 命名要"见名知意"，避免无意义的缩写（如 `u` 代替 `user`），但通用缩写（`id`、`url`、`http`）除外。
- 导出标识符（首字母大写）仅在需要跨包访问时使用，非导出标识符（首字母小写）优先。

### 2.2 各元素命名规范

- **包名**：小写、简洁、无下划线，使用单数形式，与目录名一致（如 `utils`、`service`），禁止使用复数或冗余命名（如 `userservice`）。
- **变量/函数**：非导出用小驼峰（`lowerCamelCase`），导出用大驼峰（`UpperCamelCase`）；禁止无意义缩写（如 `u` 代替 `user`），通用缩写（`id`、`url`、`http`）除外。
- **常量/结构体/接口**：
  - 枚举类常量全大写+下划线（如 `MAX_RETRY`）
  - 普通报常量大驼峰
  - 结构体大驼峰（如 `User`），避免冗余后缀（如 `UserStruct`）
  - 单个方法接口命名为"方法名+er"（如 `Reader`）

### 2.3 文件命名规范

- 小写 + 下划线分隔：`model_loader.go`、`inference_handler.go`。
- 测试文件：`*_test.go`（与被测试文件同目录）。

### 2.4 文档注释

- 使用中文注释，常见英文缩写除外。
- 所有导出元素（包、函数、类型、方法）必须有文档注释。
- 复杂逻辑添加行内注释，说明"为什么这么做"而非"做了什么"。
- 避免注释废话（如 `i++ // increment i`）。

## 3. Setup & Development（环境搭建与开发流程）

### 3.1 依赖管理

> 定期执行 `go mod tidy` 整理依赖，同步并更新所有子模块的依赖，确保所有子模块依赖一致。

- **依赖整理**：`go mod tidy`（在子模块目录执行，同步该模块独立依赖；工作空间根目录执行则同步所有子模块）
- **依赖同步**：`go work sync`（同步工作空间内所有子模块的依赖，确保一致性）
- **验证工作空间**：`go work edit -print`（查看当前工作空间配置，验证子模块关联是否正确）
- **验证依赖映射**：`go list -m all`（验证工作空间模式是否生效，确认子模块依赖指向本地而非远程）

### 3.2 项目结构

```
lego-kits/（工作空间根目录）
├── go.work     // 工作空间配置文件（核心，关联所有子模块）
├── aigc        // AI 相关工具模块
├── cgi         // 基于 gofiber/v3 的 RESTful API 服务工具模块
├── dsa         // 数据结构与通用工具方法模块
├── iam         // IAM 相关工具模块
└── oapi        // OpenAPI 相关工具模块
```

### 3.3 开发规范

- 严格验证输入维度/范围，避免无效输入。
- 导出接口而非具体类型，方便归纳通用工具方法和外部调用。
- 不忽略任何错误，使用 `fmt.Errorf("%w", err)` 包装错误链。

### 3.4 禁止事项

- 子模块内代码要符合 Go 语言规范，避免使用硬编码魔法值（如直接写 `100` 代替 `MAX_PAGE_SIZE`）。

## 4. Testing Guidelines（测试规范）

- **测试框架**：使用 `testing` 包。
- **测试文件**：与被测试文件同目录，命名为 `*_test.go`。
- **测试覆盖**：确保核心逻辑有单元测试覆盖。
